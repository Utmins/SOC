Splunk — одно из ведущих решений SIEM на рынке, которое позволяет собирать, анализировать и сопоставлять сетевые и машинные журналы в режиме реального времени.
Когда дело доходит до кибербезопасности, Splunk может играть решающую роль в качестве решения для управления журналами, но его истинная ценность заключается в его возможностях управления информацией и событиями безопасности (SIEM), основанных на аналитике.
Splunk как решение SIEM может помочь в анализе данных в реальном времени и исторических данных, мониторинге кибербезопасности, реагировании на инциденты и поиске угроз.
Более того, он позволяет организациям улучшить свои возможности обнаружения, используя аналитику поведения пользователей.

* SPLUNK Components *
*********************

Splunk имеет три основных компонента, а именно

    -   Forwarder
    -   Indexer
    -   Search Head

    +++++++++++++
    + Forwarder +
    +++++++++++++

    Splunk Forwarder — это агент, установленный на конечной точке, которую предполагается отслеживать
    Его основная задача — собирать данные и отправлять их в Indexer Splunk.
    Он не влияет на производительность конечной точки, поскольку для его обработки требуется очень мало ресурсов.
    
    Вот некоторые из основных источников данных:

        -   Веб-сервер, генерирующий веб-трафик.
        -   Машина Windows, генерирующая журналы событий Windows, PowerShell и данные Sysmon.
        -   Хост Linux, генерирующий журналы, ориентированные на хост.
        -   База данных, генерирующая запросы на подключение к БД, ответы и ошибки.
    
    Типы пересылок, используемые в Splunk:

        -   Universal Forwarder (UF)
            Это легкий агент, который собирает данные и пересылает их индексаторам Splunk без какой-либо предварительной обработки.
            Универсальные пересылки — это отдельные программные пакеты, которые можно легко установить на удаленных источниках без существенного влияния на производительность сети или хоста.

        -   Heavy Forwarder (HF)
            Этот агент служит для сбора данных из удаленных источников, особенно для интенсивной агрегации данных, включающей такие источники, как брандмауэры или точки маршрутизации/фильтрации данных.
            Тяжелые пересылки выделяются среди других типов пересылок, поскольку они анализируют данные перед пересылкой, что позволяет им маршрутизировать данные на основе определенных критериев, таких как источник или тип события.
            Они также могут индексировать данные локально, одновременно пересылая их другому индексатору.
            Обычно тяжелые пересылающие устройства развертываются как выделенные «узлы сбора данных» для доступа к данным API/скрипта, и они поддерживают исключительно Splunk Enterprise.

    *** Обратите внимание, что существуют сборщики событий HTTP (HEC), доступные для прямого сбора данных из приложений масштабируемым образом.
        HEC работают с использованием методов JSON на основе токенов или необработанных API.
        В этом процессе данные отправляются непосредственно на уровень индексатора для дальнейшей обработки.

    +++++++++++
    + Indexer +
    +++++++++++

    Индексатор Splunk играет главную роль в обработке данных, которые он получает от пересылок.
    Он берет данные, нормализует их в пары поле-значение, определяет тип данных и сохраняет их как события.
    При индексировании данных индексаторы генерируют наборы каталогов, классифицированных по возрасту, в которых каждый каталог содержит сжатые необработанные данные и соответствующие индексы, которые указывают на необработанные данные.
    Они также обрабатывают поисковые запросы от пользователей и возвращают результаты.
    Обработанные данные легко искать и анализировать.

    +++++++++++++++
    + Search Head +
    +++++++++++++++

    Splunk Search Head — это место в приложении Search & Reporting, где пользователи могут выполнять поиск в индексированных журналах.
    Когда пользователь ищет термин или использует язык поиска, известный как язык обработки поиска Splunk, запрос отправляется индексатору, и соответствующие события возвращаются в виде пар «поле-значение».
    Search Head также предоставляет возможность преобразовывать результаты в презентабельные таблицы и визуализации, такие как круговые диаграммы, гистограммы и столбчатые диаграммы

    Поисковые заголовки координируют поисковые задания, отправляя их индексаторам и объединяя результаты.
    Они также предоставляют интерфейс для взаимодействия пользователей со Splunk.
    В Search Heads объекты знаний могут быть созданы для извлечения дополнительных полей и манипулирования данными без изменения исходных данных индекса.
    Важно отметить, что Search Heads также предлагает различные инструменты для обогащения опыта поиска, включая отчеты, панели мониторинга и визуализации.

Если вы используете Splunk Enterprise, то Вам будут доступные следующие основные компоненты:

    +++++++++++++++++++++
    + Deployment Server +
    +++++++++++++++++++++

    Сервер развертывания управляет конфигурацией для пересылок, распространения приложений и обновлений

    ++++++++++++++++++
    + Cluster Master +
    ++++++++++++++++++

    Мастер кластера координирует действия индексаторов в кластерной среде, обеспечивая репликацию данных и соответствие поиска.

    ++++++++++++++++++
    + License Master +
    ++++++++++++++++++

    Мастер лицензий управляет лицензионными подробностями платформы Splunk.

* Adding Data *
***************

Splunk может принимать любые данные.
Согласно документации Splunk, когда данные добавляются в Splunk, они обрабатываются и преобразуются в серию отдельных событий.

Источниками данных могут быть журналы событий, журналы веб-сайтов, журналы брандмауэров и т. д.
Источники данных сгруппированы по категориям.
Ниже приведен список диаграмм из документации Splunk, в котором подробно описана каждая категория источников данных.

    Data source	                Description
    Files and directories       Большинство данных, которые могут вас заинтересовать, поступают напрямую из файлов и каталогов.
    Network events              Программное обеспечение Splunk может индексировать удаленные данные с любого сетевого порта и события SNMP с удаленных устройств.
    IT Operations               Данные из ИТ-операций, таких как Nagios, NetApp и Cisco.
    Cloud services              Данные из облачных служб, таких как AWS и Kinesis.
    Database services           Данные из баз данных, таких как Oracle, MySQL и Microsoft SQL Server.
    Security services           Данные из служб безопасности, таких как McAfee, Microsoft Active Directory и Symantec Endpoint Protection.
    Virtualization services     Данные из служб виртуализации, таких как VMWare и XenApp.
    Application servers         Данные из серверов приложений, таких как JMX и JMS, WebLogic и WebSphere.
    Windows sources             Версия программного обеспечения Splunk для Windows принимает широкий спектр входных данных, специфичных для Windows, включая журнал событий Windows, реестр Windows, WMI, Active Directory и мониторинг производительности.
    Other sources               Поддерживаются другие источники входных данных, такие как очереди FIFO и скриптовые входные данные для получения данных из API, а также другие удаленные интерфейсы данных.

Для загрузки данных с нашего локального компьютера необходимо воспользоваться опцией Upload. Загрузите прикрепленный файл журнала и загрузите его на Splunk.
Для успешной загрузки данных требуется всего 5 шагов.

    1)  Select Source       ->  Где мы выбираем Источник журнала.
    2)  Select Source Type  ->  Выберите тип журналов, которые будут приниматься.
    3)  Input Settings      ->  Выберите индекс, куда будут выгружаться эти журналы, и имя хоста, которое будет связано с журналами.
    4)  Review              ->  Просмотр всех gif
    5)  Done                ->  Последний шаг, где данные успешно загружены и готовы к анализу

Существует НАМНОГО больше журналов, которые мы можем добавить в экземпляр Splunk, и Splunk поддерживает различные типы источников.

* Incedent Handling *
*********************

Incident с точки зрения безопасности — это «Любое событие или действие, которое имеет негативные последствия для безопасности пользователя/компьютера или организации, считается инцидентом безопасности».
Ниже приведены некоторые события, которые могут негативно повлиять на окружающую среду, если они произошли:

    -   Сбой системы
    -   Выполнение нежелательной программы
    -   Доступ к конфиденциальной информации от неавторизованного пользователя
    -   Взлом веб-сайта злоумышленником
    -   Использование USB-устройств при наличии ограничений на использование противоречит политике компании

Как Incident Handler / SOC Analyst, мы стремимся узнать тактику, методы и процедуры атакующих.
Тогда мы сможем остановить/защитить/предотвратить атаку лучшим способом.
Процесс обработки инцидентов делится на четыре различных этапа

    1)  Preparation (Подготовка)

        Фаза подготовки охватывает готовность организации к атаке.
        Это означает документирование требований, определение политик, включение элементов управления безопасностью для мониторинга, таких как EDR / SIEM / IDS / IPS и т. д.
        Она также включает в себя найм/обучение персонала.
    
    2)  Detection and Analysis (Обнаружение и анализ)

        Фаза обнаружения охватывает все, что связано с обнаружением инцидента и процессом анализа инцидента.
        Эта фаза охватывает получение оповещений от элементов управления безопасностью, таких как SIEM/EDR, расследование оповещения для поиска первопричины.
        Она также охватывает поиск неизвестной угрозы внутри организации.
    
    3)  Containment, Eradication, and Recovery (Сдерживание, ликвидация и восстановление)

        Данная фаза охватывает действия, необходимые для предотвращения распространения инцидента и обеспечения безопасности сети.
        Она включает шаги, предпринимаемые для предотвращения распространения атаки в сети, изоляции зараженного хоста, очистки сети от следов заражения и восстановления контроля над атакой.
    
    4)  Post-Incident Activity / Lessons Learnt (Действия после инцидента/Извлеченные уроки)

        Эта фаза включает в себя выявление лазеек в состоянии безопасности организации, которые привели к вторжению, и улучшение, чтобы атака не повторилась в следующий раз.
        Шаги включают в себя выявление слабых мест, которые привели к атаке, добавление правил обнаружения, чтобы подобное нарушение не повторилось, и, что самое важное, обучение персонала, если это необходимо.

* Leverage Splunk's User Interface for Data & field identification *
********************************************************************

В любой надежной системе управления информацией о безопасности и событиями (SIEM), такой как Splunk, понимание доступных источников данных, данных которые они предоставляют, и полей в них имеет решающее значение для эффективного использования системы.
В Splunk мы в первую очередь используем приложение Search & Reporting для этого.

При использовании пользовательского интерфейса приложения Search & Reporting определение доступных типов источников данных, содержащихся в них данных и полей, становится задачей, которая включает взаимодействие с различными разделами пользовательского интерфейса.
Давайте рассмотрим, как мы можем эффективно использовать веб-интерфейс Splunk для определения этих элементов.

    -   Data Sources
        Первое, что мы хотим определить, — это наши источники данных.
        Мы можем сделать это, перейдя в меню «Settings».
        В раскрывающемся списке мы найдем Data inputs.
        Нажав на «Data inputs», мы увидим список различных методов ввода данных, включая файлы и каталоги, сборщик событий HTTP, пересылки, скрипты и многое другое.
        Они представляют собой различные источники, через которые данные могут быть введены в Splunk.
        Нажатие на каждый из них даст нам обзор используемых источников данных.

    -   Data (Events)
        Теперь давайте перейдем к определению самих данных (data), другими словами, событий (events).
        Для этого нам нужно перейти в приложение «Search & Reporting ».
        Исследуя события в быстром режиме (Fast mode), мы можем быстро просмотреть наши данные
        С другой стороны, режим Verbose позволяет нам глубоко погрузиться в детали каждого события, включая его необработанные данные о событиях и все поля, которые Splunk извлек из них.

        В строке поиска мы могли бы просто поставить * и нажать «search», что выведет все данные, которые мы проиндексировали.
        Однако обычно это огромный объем данных, и это может быть не самым эффективным способом.
        Лучшим подходом может быть использование выбора времени и выбор меньшего временного диапазона (следует быть осторожным, делая это, чтобы не пропустить какие-либо важные/полезные исторические журналы).
    
    -   Fields
        Наконец, чтобы определить поля в наших данных, мы можем нажать на любое событие в результатах поиска, чтобы развернуть его.
        Мы также можем видеть в левой части приложения «Search & Reporting» две категории полей:
            
            Selected Fields
            Выбранные поля — это поля, которые всегда отображаются в событиях (например, хост, источник и тип источника),

            Interesting Fields
            Интересные поля — это те, которые появляются как минимум в 20% событий.
        
        Нажав All fields, мы можем увидеть все поля, присутствующие в наших событиях.

    -   Data Models
        Модели данных обеспечивают организованное, иерархическое представление наших данных, упрощая сложные наборы данных до понятных структур.
        Они разработаны для упрощения создания содержательных отчетов, визуализаций и панелей мониторинга без необходимости глубокого понимания базовых источников данных или необходимости написания сложных запросов.
        Вот как мы можем использовать настрокиу Data Models для идентификации и понимания наших данных:

            Accessing Data Models
            Чтобы получить доступ к моделям данных, мы нажимаем на вкладку Settings в правом верхнем углу веб-интерфейса Splunk.
            Затем мы выбираем Data Models в разделе Knowledge.
            Это приведет нас на страницу управления моделями данных.
            *** Если она пуста, выполните поиск и снова перейдите на страницу Data Models.

            Understanding Existing Data Models
            На странице Data Models management мы можем увидеть список доступных моделей данных. 
            Они могут включать модели, созданные нами, нашей командой, или модели, предоставленные приложениями Splunk.
            Каждая модель данных связана с определенным контекстом приложения и создана для описания структурированных данных, связанных с целью приложения.

            Exploring Data Models
            Нажавна на Name of a Data Model, мы попадаем в Data Model Editor.
            Именно здесь и заключается истинная сила моделей данных.
            Здесь мы можем просмотреть иерархическую структуру модели данных, которая разделена на объекты.
            Каждый объект представляет определенную часть наших данных и содержит поля, которые имеют отношение к этому объекту.
            Например, если у нас есть модель данных, описывающая веб-трафик, мы можем увидеть такие объекты, как веб-транзакции, веб-ошибки и т. д.
            Внутри этих объектов мы найдем такие поля, как статус, URL, пользователь и т. д.

    -   Pivots (Сводки)
        Сводки — это чрезвычайно мощная функция в Splunk, которая позволяет нам создавать сложные отчеты и визуализации без написания запросов
        Они предоставляют интерактивный интерфейс с функцией перетаскивания для определения и уточнения наших критериев отчетности по данным.
        Таким образом, они также являются фантастическим инструментом для идентификации и изучения доступных данных и полей в нашей среде Splunk.
        Чтобы начать работу с Pivots для определения доступных данных и полей, мы можем использовать кнопку Pivot, которая появляется при просмотре определенной модели данных на странице «Data Models».

* Splunk Applications *
***********************

Приложения Splunk (Splunk Applications) -   это пакеты, которые мы добавляем в наши развертывания Splunk Enterprise или Splunk Cloud для расширения возможностей и управления определенными типами рабочих данных.
Каждое приложение предназначено для обработки данных из определенных технологий или вариантов использования, эффективно выступая в качестве предварительно созданного пакета знаний для этих данных.
Приложения могут предоставлять возможности, начиная от настраиваемых входных данных, настраиваемых визуализаций, панелей мониторинга, оповещений, отчетов и многого другого.
Они предназначены для приема, анализа и визуализации данных, связанных с безопасностью, что позволяет нам обнаруживать сложные угрозы и проводить углубленные расследования.

Проще говоря Splunk Applications (приложения SPLUNK)    -   являются набор взаимосвязанных компонентов: дашбордов, форм, отчётов, конфигураций, настроек, алертов, и иногда собственных скриптов.
Которые можно воспринимать как мини-приложение внутри Splunk.

Вот основные особенности Splunk Applications:

    -   Объединяет несколько дашбордов, конфигурации полей, тэги, knowledge objects.

    -   Может содержать настройки парсинга, props/transforms, lookup-файлы, REST-интеграции и даже Python-скрипты.

    -   Может быть создан пользователем или загружен из Splunkbase (например, Cisco Security Suite, Splunk App for Windows Infrastructure).

При использовании этих приложений в среде Splunk нам необходимо учитывать такие факторы, как объем данных, требования к оборудованию и лицензирование.
Многие приложения могут быть ресурсоемкими, поэтому мы должны убедиться, что наше развертывание Splunk имеет правильный размер для обработки дополнительной рабочей нагрузки.
Кроме того, также важно убедиться, что у нас есть правильные лицензии для любых премиум-приложений, и что мы осознаем потенциал увеличения использования лицензий из-за дополнительных входных данных.

Приложения Splunk обеспечивают сосуществование нескольких рабочих пространств на одном экземпляре Splunk, обслуживая различные варианты использования и роли пользователей.
Эти готовые приложения можно найти на Splunkbase (https://splunkbase.splunk.com/)

Существует несколько вариантов использования SPLUNK Applications
При этом каждое из которых имеет свои уникальные особенности

    ++++++++++++++++++++
    + Install app from +
    ++++++++++++++++++++

    Данный способ хорош тем, что Вам не нужно заморачиваться с созданием App с нуля
    Достаточно внедрить уже заранее скачаное приложение
    Тем не менее потребуется определенная настрока скачанного App, чтобы оно функционировало должным образом

    Рассмотрим на примере SYSMON
    Чтобы загрузить, добавить и использовать необходимое приложение, выполните следующие действия:

        1)  Зарегистрируйте бесплатную учетную запись на splunkbase

        2)  После регистрации войдите в свою учетную запись

        3)  Перейдите на страницу Sysmon App for Splunk, чтобы загрузить/скачать необходимое приложение.

        4)  Перейдите в сам SPLUNK

        5)  Добавьте приложение в строку поиска следующим образом:

                -   на главной странице/экране SPLUNK, в левой верхней части рабочей области найдите значок "шестеренка" (располагается напротив надписи Apps) и нажмите на него
                -   в появившемся окне, в правой верхней части рабочего области, найдите кнопку "Install app from the"
                -   выберете скачаное приложение
        
        6)  После завершения установки и возможной перезагрузки SPLUNK, вернитесь на главную рабочую область SPLUNK.
            Убедитесь, что в левой части рабочего области (среди списка быстрого дустпа к некоторым Apps) отображается, только что устанровленое новое Apps
        
        7)  Далее необходимо настроить макрос скачанного приложения так, чтобы события загружались должным образом образом.
            Для этого в правой части главное рабочей области перейдите в Settings -> Advence Search -> Search Macros
        
        8)  В появившемся окне, в выпадающем меню  (в левой верхней части окна) выбирите необходимое приложение
            В отфильтрованном результате, нажмите на имя приложения
        
        9)  Отредактируйте поле Defenition
            На примере Sysmon это долджно выглядеть так

                Из sourcetype="XmlWinEventLog:Microsoft-Windows-Sysmon/Operational" в index="main" sourcetype="WinEventLog:Sysmon"
            
            Но кажое приложение может иметь свои индивидуальные настройки
        
        10) Вернитесь на главную рабочую область и передите в установленное прилодение, выбрав его из списка быстродоступных приложений (в левой части окна)

        11) В появившемся окне, среди спика доступных опций (врехня часть окна) найдите File Activity
            Нажмите на него и выбирите File Creation Overview
            
        12) В появившемся окне выбирите необходимый временной диапозон и нажмите Submit

        13) Если среди полученных результатов, какая-либо из секций не будет содержать данных/результатов, то в правом верхнем углу рабочего окна найдите кнопку Edit и нажмите ее
            Далее, прокрутите экран вниз до необходимой секции и нажмите на значок "Лупа" даннйо секции

        14) Должно появиться окно "Edit Search"
            Отредактируйте его должным образом
            К примеру   -   События Sysmon с ID 11 не содержат поля с именем Computer, но они включают поле с именем ComputerName.
                            Поуэтому надо будет заменить слово Computer на ComputerName
            Каждое приложение и/или секций приложения может иметь поля с уникальными для нее названиями

            *** Не стесняйтесь исследовать и экспериментировать.
                Отличным упражнением будет изменение поисков, когда результаты не генерируются из-за указания несуществующих полей, и продолжение до тех пор, пока не будут получены желаемые результаты.
    
    +++++++++++++++++++++++++++++++++++++++
    + Creating Custom Splunk Applications +
    +++++++++++++++++++++++++++++++++++++++

    Также может потребоваться создать собственное App с нуля
    Для этого Вам потребуется Web интерфейс SPLUNK

    Итак, вот что нужно сделать

        1)  Access Splunk Web
            Откройте веб-браузер и перейдите в Splunk Web.
        
        2)  Go to Manage Apps
            В строке меню вверху выберите «Apps», а затем «Manage Apps».
        
        3)  Create a New App
            На странице «Apps» нажмите «Create app» (находится в правой верхней части рабочей области)
        
        4)  Enter App Details
            На появившейся странице «Add new» заполните свойства нового приложения:

                -   Name
                    Введите имя вашего App (произвольное).

                -   Folder name
                    Укажите имя папки, которое должно быть похоже на имя вашего App, но не обязательно (главное, чтобы было понятно Вам).
                    Оно будет располагаться в каталоге $SPLUNK_HOME/etc/apps/

                -   Version
                    Для впервые создаваемого App рекомендуется вводить "1.0.0".

                -   Description
                    Укажите описание вашего App (произвольно, но чтобы отражало характер действий)

                -   Template
                    В большинстве случаем рекомендуется использовать шаблок "barebones" (из раскрывающегося меню).
                    Однако, если Вы знакомы с другими представленными шабонами (из выпадаюшего меню), то можете использовать один из них
        
        5)  Save the App
            Нажмите «Save».
            Вы можете убедиться, что ваше приложение было создано, перейдя в меню « App».
            Ваше новое приложение теперь должно быть там указано.
            Кроме того, если вы перейдете на домашнюю страницу Splunk Web, вы найдете свое  App в списке приложений.
        
        6)  Explore the Directory Structure
            Чтобы ознакомиться с содержанием и структурой каталога, где было сохранено Ваше созданное App, необходимо использовать файловый браузер для перехода в $SPLUNK_HOME/etc/apps.
            Здесь вы найдете каталог вашего приложения, который включает следующие папки:

                -   /bin
                    Здесь хранятся скрипты.

                -   /default
                    В этом каталоге хранятся файлы для конфигурации, представлений, панелей мониторинга и навигации по приложениям.

                -   /local
                    В этом каталоге содержатся измененные пользователем версии файлов для конфигурации, представлений, панелей мониторинга и навигации по приложениям.

                -   /metadata
                    В этом каталоге хранятся файлы разрешений.
        
        7)  View the Configuration (Navigation) File
            Файл конфигурации (навигации)   -   это XML-файл, который содержит основне настрокий вашего App
            
            Чтобы просмотреть (и при необходимости отредактировать) его содержимое, можно использовать 2 следующих способа:
            
                a)  Используя текстовый редактор, откройте $SPLUNK_HOME/etc/apps/<your app>/default/data/ui/nav/default.xml

                b)  Access Splunk Web -> Go to Manage Apps -> Find you App -> Click on "View Objects" (находитя в той же строке, что и Ваш App, но в самом конце) -> Click on "defaults"
            
            Только,что созданное App (т.е. без каких-либо изменений/дополнений) будет иметь следующее содержанием

                <nav search_view="search">
                    <view name="search" default='true' />
                    <view name="analytics_workspace" />
                    <view name="datasets" />
                    <view name="reports" />
                    <view name="alerts" />
                    <view name="dashboards" />
                </nav>
            
            Давайте разберем его составлющие элемнеты:

                -   nav
                    Данный ТЭГ верхнего уровня выступает в качестве родителя.
                
                -   search_view
                    Этот атрибут обозначает представление по умолчанию для поиска.
                    В этом случае используется представление поиска, унаследованное от приложения Search & Reporting.
                
                    Все идущее ниже элементы/тэги (в иерархии XML) соответствуют элементам, отображаемым на панели приложения.
                    Список тегов обозначает различные представления для отображения, где каждое из представлений соответствует представлению из приложения Search & Reporting.
                
                -   default='true'
                    Данный атрибут указывает представление, которое будет использоваться в качестве домашней страницы приложения — здесь представление поиска служит домашней страницей.

* Splunk Dashboard *
********************

Как Вы уже поняли, Splunk Dashboard является одним из компонентов Splunk Applications
Это своего рода визуальное представление данных, собранных из Splunk Search.
Он состоит из панелей (panels) с графиками, таблицами, гистограммами и т.п., и используется для мониторинга, анализа и отчётности.

    Вот основные особенности Splunk Dashboard:

    -   Может быть частью одного или нескольких приложений (Apps).

    -   Основан на XML, SimpleXML или Dashboard Studio (JSON/YAML).

    -   Обычно делает одно  -   показывает данные в удобной форме.

    -   Можно создать быстро, даже без программирования.

Проще говоря, Splunk Dashboard используется если нужно просто визуализировать конкретный набор данных.

Для создания своего собственного Splunk Dashboard нужно выполнить следующие шаги:

    1)  Create Your Dashboard
        Click on "Apps" -> Search & Reporting -> Click on "Dashboards"
        B нажмите «Create New Dashboard».
        Введите"
        
            -   Title
                Имя панели мониторинга
            
            -   Description
                При необходимости предоставьте описание
            
            -   Permissions
                Выберите «Shared in App» и установите разрешения на "Classic Dashboards"
    
    2)  Configure the Dashboard
        Теперь вы увидите страницу редактора панели мониторинга, где вы можете настроить панели, входы и т. д. для упрощения процесса мониторинга.
        
            -   Добавьте "time input" (входные данные времени) для панели мониторинга и настройте "time range" (диапазон времени) по умолчанию в соответствии с вашими потребностями.
        
            -   Затем добавьте  "statistical table panel" (панель статистической таблицы), выберите диапазон времени для "Shared Time Picker", добавьте "Content Title" (заголовок контента, например, "Sysmon Process Events") и введите "Search String" (строку поиска).
        
            -   Чтобы использовать входные данные в поиске, заключите токен ввода в знаки доллара, например $user$.
       
            -   Нажмите Add to Dashboard, когда будете готовы. Сохраните изменения.
    
    3)  Dashboard Storage
        Все панели мониторинга, созданные в вашем приложении, хранятся в "<AppPath>/local/data/ui/views/dashboard_title.xml".
        Чтобы добавить панель мониторинга в панель навигации, просто добавьте заголовок панели мониторинга в XML-код страницы навигации по умолчанию: "<AppPath>/local/data/ui/nav/default.xml".
    
    4)  Restart Splunk
        Чтобы только что созданная "Splunk Dashboard" начала работать, необходимо перезагрузите экземпляр Splunk.
        После перезапуска вы должны увидеть свою панель управления на панели навигации.
    
    5)  Grouping Dashboards
        Если вы хотите сгруппировать несколько панелей мониторинга под одной записью в панели навигации, используйте тег коллекции.

            <collection label="<Collection Name">
                <view name="<first_dashboard_name" />
                <view name="<second_dashboard_name" />
            <collection />
